# Orbit Wars Notes

## 游戏规则

这一节按 Kaggle 比赛页 `How to Play Orbit Wars` 的内容展开，不再只做摘要，而是把那一页里的规则点完整转写成中文说明。

来源页面：

- https://www.kaggle.com/competitions/orbit-wars/overview

### 总览

玩家开局各自拥有一颗母星，目标是通过发射舰队占领中立星球和敌方星球，从而控制整张地图。

地图是一个 `100 x 100` 的连续二维空间，中间有太阳。星球会围绕太阳旋转，彗星会沿着高椭圆轨迹穿过棋盘，舰队则沿直线飞行。整局游戏一共进行 `500` 个回合。游戏结束时，拥有总舰船数最多的玩家获胜。这里的总舰船数等于：

- 己方所有星球上的驻军
- 己方所有舰队中的舰船

### 棋盘布局

- 棋盘大小是 `100 x 100`
- 坐标原点在左上角
- 太阳中心在 `(50, 50)`
- 太阳半径是 `10`
- 舰队如果路径穿过太阳，就会被摧毁

地图中的星球和彗星都按四向镜像对称放置。对称点为：

- `(x, y)`
- `(100 - x, y)`
- `(x, 100 - y)`
- `(100 - x, 100 - y)`

这种四重对称是为了保证不同出生位置的公平性。

### 星球

每颗星球在观测中的格式是：

```python
[id, owner, x, y, radius, ships, production]
```

字段含义如下：

- `id`: 星球唯一编号
- `owner`: 所属玩家，取值 `0` 到 `3`，若为 `-1` 则表示中立星球
- `x, y`: 当前坐标
- `radius`: 星球半径
- `ships`: 当前驻军数
- `production`: 每回合产兵数

关于星球属性还有以下规则：

- `radius` 由 `production` 决定，公式是 `1 + ln(production)`
- 因此高产星球会更大
- `production` 是 `1` 到 `5` 的整数
- 只要星球有主人，它每回合就会额外生成这么多船
- `ships` 表示当前守军
- 初始驻军在 `5` 到 `99` 之间，但分布更偏向较小值

### 星球类型

Kaggle 页面把星球分成两类：

- 旋转星球
- 静止星球

旋转星球的判定条件是：

- `orbital_radius + planet_radius < 50`

也就是星球轨道半径和星球半径之和小于 `50`

旋转星球会以固定角速度绕太阳旋转。该角速度在每局开始时随机确定，范围是：

- `0.025` 到 `0.05` 弧度/回合

如果要预测它们的未来位置，Kaggle 页面明确提示可以使用观测里的：

- `initial_planets`
- `angular_velocity`

静止星球则是那些离中心更远、不会旋转的星球。

### 地图中的星球数量

整张地图通常包含：

- `20` 到 `40` 颗星球

也可以理解为：

- `5` 到 `10` 组对称星群
- 每组有 `4` 颗镜像对应的星球

此外，页面还写明：

- 至少会有 `3` 组静止星球
- 至少会有 `1` 组旋转星球

### 出生星球

游戏开始时，会随机挑选一组对称星球作为出生点。

在 `2` 人对局中：

- 双方出生在对角线相对的两颗星球上
- 页面里写的是第一象限和第四象限对应的出生点

在 `4` 人对局中：

- 这一组中的四颗星球会分别给四名玩家各一颗

所有出生星球初始都有：

- `10` 艘船

### 舰队

舰队在观测中的格式是：

```python
[id, owner, x, y, angle, from_planet_id, ships]
```

字段含义如下：

- `id`: 舰队编号
- `owner`: 所属玩家
- `x, y`: 当前坐标
- `angle`: 飞行方向，单位是弧度
- `from_planet_id`: 这支舰队是从哪颗星球发出的
- `ships`: 这支舰队包含多少艘船

页面还特别说明：

- `ships` 在飞行途中不会变化

### 舰队速度

舰队速度随舰船数量变化，而且是对数型增长。Kaggle 页面给出的公式是：

```python
speed = 1.0 + (maxSpeed - 1.0) * (log(ships) / log(1000)) ^ 1.5
```

页面给出的具体解释包括：

- `1` 艘船时，速度是 `1.0`
- 船数越多，速度越快
- 速度逐渐逼近最大速度
- 默认最大速度是 `6.0`
- 大约 `500` 艘船时，速度接近 `5`
- 大约 `1000` 艘船时，可以达到最大速度

### 舰队移动

舰队每回合会沿当前方向做直线运动，移动距离由上面的速度公式决定。

一支舰队会在以下任意情况发生时被移除：

- 飞出边界，也就是离开 `100 x 100` 的可玩区域
- 路径穿过太阳
- 路径撞到任意星球

当舰队撞上星球时，会触发战斗。

页面特别强调，碰撞检测是连续的，而不是只看新位置终点。也就是说：

- 会检查舰队从旧位置到新位置的整条线段
- 不是只判断移动后终点有没有落到圆内

### 舰队发射

每回合 agent 要返回一个动作列表。每个动作格式是：

```python
[from_planet_id, direction_angle, num_ships]
```

Kaggle 页面列出的发射规则包括：

- 只能从己方拥有的星球发射
- 不能发出超过该星球当前驻军数量的船
- 新舰队会生成在星球半径之外、沿指定方向刚刚离开星球表面的位置
- 同一回合里，可以从同一颗星球或不同星球发出多支舰队

### 彗星

彗星是临时出现的天体，Kaggle 页面把它定义为沿太阳附近的高椭圆轨迹穿过棋盘的额外天体。

它们有以下特性：

- 每次以 `4` 个为一组生成
- 每个象限各出现一个
- 会在第 `50`、`150`、`250`、`350`、`450` 回合生成

彗星的参数如下：

- 半径固定为 `1.0`
- 被占领后每回合产兵 `1`
- 初始舰船数是随机的，但偏向小值
- 初始舰船数由 `1` 到 `99` 的四次随机抽样取最小值得到
- 同一组里的 `4` 颗彗星共享同一个初始舰船数
- 彗星速度由 `cometSpeed` 控制
- 默认值是 `4.0` 单位/回合

识别彗星的方式：

- 在观测中查看 `comet_planet_ids`
- 这些 ID 对应的天体就是彗星

同时页面也强调：

- 彗星同样会出现在 `planets` 数组里
- 它们遵守普通星球的规则
- 可以被占领
- 会产兵
- 可以从上面发射舰队
- 也会参与战斗

关于彗星移除规则，页面写得也很明确：

- 当彗星飞出棋盘后，它会被直接删除
- 同时删除驻扎在上面的所有舰船
- 每回合开始时会先移除已离场的彗星
- 因此你不能从本回合正要离场的彗星上发兵

关于观测，页面还提到：

- `comets` 字段里会保存彗星组的信息
- 其中包括 `paths`，也就是每颗彗星整条轨迹
- 以及 `path_index`，表示当前走到轨迹上的哪个位置

这些数据可以用于预测未来的彗星位置。

### 回合执行顺序

Kaggle 页面列出的完整回合顺序是：

1. 先移除已经离开棋盘的彗星
2. 在指定回合生成新的彗星组
3. 处理所有玩家动作，创建新舰队
4. 所有已拥有的星球开始产兵，彗星也一样
5. 所有舰队按当前方向移动，并检查越界、穿太阳、撞星球
6. 旋转星球继续公转，彗星沿轨迹推进；如果移动中的星球或彗星扫到舰队，也会把舰队卷入战斗
7. 结算所有排队中的星球战斗

这份顺序比常见摘要版更细，其中前两步是专门给彗星留出的：

- 先过期删除
- 再按时间点生成

### 战斗规则

当一支或多支舰队撞上某颗星球时，不管是主动飞进去，还是被移动中的星球/彗星扫到，都会触发战斗。

页面把战斗分成下面几步：

1. 所有到达该星球的舰队，先按所有者分组
2. 同一所有者的舰船数先求和
3. 最大攻击方与第二大攻击方互相作战，差值留下来
4. 如果有攻击方存活，再拿这些幸存舰船去和星球驻军结算

接着按幸存方和星球归属来分情况：

- 如果幸存攻击方和星球是同一所有者，则幸存舰船加入驻军
- 如果幸存攻击方和星球所有者不同，则幸存舰船和原驻军相减
- 如果攻击方数量超过驻军，则星球易主
- 易主后新的驻军数量等于多出来的那部分舰船

页面还额外说明了平局情况：

- 如果两个攻击方并列第一
- 那么所有攻击舰队都被消灭
- 不会有攻击方幸存

### 终局与计分

游戏会在以下两种条件之一满足时结束：

- 回合数达到上限，也就是 `500` 回合
- 场上只剩一个玩家，或者没有任何玩家还拥有星球或舰队

最终分数的定义是：

- 所有己方星球驻军之和
- 加上所有己方舰队舰船数之和

总数最高者获胜。

### 观测字段参考

Kaggle 页面给出了一张观测字段对照表，这里完整转写成中文：

| 字段 | 类型 | 含义 |
|------|------|------|
| `planets` | `[[id, owner, x, y, radius, ships, production], ...]` | 当前所有星球，包含彗星 |
| `fleets` | `[[id, owner, x, y, angle, from_planet_id, ships], ...]` | 当前所有活动舰队 |
| `player` | `int` | 当前玩家自己的 ID，范围 `0` 到 `3` |
| `angular_velocity` | `float` | 星球旋转角速度，单位是弧度/回合 |
| `initial_planets` | `[[id, owner, x, y, radius, ships, production], ...]` | 游戏开始时的星球位置 |
| `comets` | `[{planet_ids, paths, path_index}, ...]` | 当前活跃彗星组的信息 |
| `comet_planet_ids` | `[int, ...]` | 哪些 planet ID 是彗星 |
| `remainingOverageTime` | `float` | 剩余额外时间预算，单位秒 |

### 动作格式

页面单独再次说明了动作返回格式：

```python
[[from_planet_id, direction_angle, num_ships], ...]
```

其中：

- `from_planet_id`: 你拥有的某颗星球 ID
- `direction_angle`: 弧度角，`0` 表示向右，`pi / 2` 表示向下
- `num_ships`: 发出的舰船数量，整数

如果本回合不想行动，可以返回：

```python
[]
```

### 代码层面的便利工具

Kaggle 页面还给了一个“agent convenience”段，意思是环境模块已经导出了便于使用的命名元组和常量。

可直接导入：

```python
from kaggle_environments.envs.orbit_wars.orbit_wars import Planet, Fleet, CENTER, ROTATION_RADIUS_LIMIT
```

其中：

- `Planet` 便于按属性访问星球字段
- `Fleet` 便于按属性访问舰队字段
- `CENTER` 是地图中心
- `ROTATION_RADIUS_LIMIT` 是旋转圈与静止圈的分界半径相关常量

### 环境配置

页面最后给出了默认配置表：

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `episodeSteps` | `500` | 最大回合数 |
| `actTimeout` | `1` | 每回合动作时间限制，单位秒 |
| `shipSpeed` | `6.0` | 舰队最大速度 |
| `sunRadius` | `10.0` | 太阳半径 |
| `boardSize` | `100.0` | 棋盘边长 |
| `cometSpeed` | `4.0` | 彗星移动速度 |
